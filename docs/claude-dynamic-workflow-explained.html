🌱 一分钟理解：什么是「动态工作流」？

想象一个场景：你有一个大项目需要完成，比如审查一份 100 页的代码变更。

传统方式：你一个人从头到尾逐页审查 — 很慢，而且你容易疲劳遗漏。

动态工作流方式：你写一个「任务分配清单」，列出需要检查的维度（安全性、性能、风格…），然后把每个维度同时分配给不同的专家。每位专家独立工作，完成后把结果汇总给你。你不需要等所有人全部做完 — 有专家先完成就先看，全部完成后你做最终检查。

这就是动态工作流：Claude 自动编写任务分配脚本，同时派出多个「子专家」（我们叫它「子智能体」），并行处理，最后汇总检查。关键创新是 — 这个分配清单是 Claude 实时编写的，不是提前固定的模板。

技术概述

2026 年，Anthropic 在 Claude Code 中发布了 动态工作流（Dynamic Workflows） 功能。其核心理念是：Claude 动态编写 JavaScript 编排脚本，在单次会话中扇出（fan-out）数十到数百个并行子智能体，在将任何结果呈现给用户之前就完成自我检查和验证。

然而，官方公告仅提供了概念性描述 — 没有 API、没有函数名、没有日志格式。官方文档确认了约束条件，但至今未公开脚本 API。那么，我们如何理解其内部机制？

§3 核心架构：五大机制

一个"动态工作流"本质上是 Claude 编写的普通 JavaScript 脚本；引擎是一个执行该脚本并将每个 agent() 调用转化为真实的、可调度的、可恢复的子智能体的运行时。以下五大机制完成了所有工作。

🔒 机制一：脚本即数据，沙箱执行

编排脚本在 node:vm 沙箱中运行，其全局变量只有注入的原语加上少数安全的内置对象：

📚 关键设计意图

时间戳和随机函数被替换为抛出异常的桩函数 — 不是为了安全，而是为了确定性。恢复（resume）机制对每次调用的输入进行哈希，如果脚本中有活时钟或随机数生成器，回放时缓存键就会改变。

🔐 安全注意事项

node:vm 沙箱不是安全边界 — 注入的宿主对象（console、全局变量甚至 agent）通过 .constructor 暴露了宿主领域的 Function，因此工作流脚本可以访问 process。仅运行受信任的工作流脚本（在真实引擎中，脚本由 Claude 编写，因此是可信的）。

⚡ 机制二：agent() 是原子操作

每个 agent() 调用遵循完全相同的流程。在任何 await 之前同步地获取其在序列中的位置，因此调用顺序是确定性的：

// 以下伪代码展示 agent() 的完整执行流程

// 步骤 1: 同步分配调用序号（在任何 await 之前）
const ordinal = ++state.ordinal          // 确定性的调用位置

// 步骤 2: 计算前缀链式哈希
const { key, chain } = chainKey(state.chain, prompt, opts)
state.chain = chain                      // 推进链给下一次调用

// 步骤 3: 生成确定性 agentId
const agentId = "a" + String(ordinal).padStart(3, "0")
  + "-" + (++state.agentSeq)

// 步骤 4: 在获取并发槽位之前检查缓存
const cached = journal.lookup(key)
if (cached.hit) return cached.result    // 恢复：跳过后端和限流器！

// 步骤 5: 通过并发限流器执行
await limiter.run(async () => {
  budgetCeilingCheck()              // 输出 token 上限检查
  if (agentCount >= 1000) throw  // 生命周期上限

  journal.recordStarted(key, agentId)

  // ⬇️ 真实智能体在这里运行！
  const resp = await backend.run({
    prompt, schema, model, agentId
  })

  state.tokensSpent += resp.outputTokens
  journal.recordResult(key, agentId, resp.output)
  return resp.output
})
✓ 关键设计点

缓存查找在 limiter.run 之前 — 完全缓存的恢复完全绕过并发槽位，不会串行化通过限流器，进度遥测也能正确报告命中。

🔐 机制三：前缀链式哈希恢复

每个活跃的 agent() 调用向 journal.jsonl 追加 {started} 然后 {result, key}。重新运行脚本时，每个键被重新计算；匹配时直接返回存储的结果，不调用后端。

缓存键不是简单的 hash(prompt)，而是：

// 前缀链式缓存键算法（ANALYSIS §6 + 实验六验证）
// key_n 混合了运行中的 chain 摘要（之前所有调用的信息）+ 当前调用的 prompt + opts

function chainKey(prevChain, prompt, opts) {
  const optsDigest = stableStringify({
    schema: opts.schema ?? null,
    model: opts.model ?? null,
    agentType: opts.agentType ?? null,
    isolation: opts.isolation ?? null,
  })
  const chain = sha256(prevChain + " " + prompt + " " + optsDigest)
  return { key: "v2:" + chain, chain }
}

级联效应（实验 06 证明）

设计了一个关键实验：4 个独立顺序智能体 A → B → C → D，然后仅编辑 B 并恢复。

A
✓ 缓存命中
➔
B
✗ 已编辑，重跑
➔
C
✗ prompt 未变，但 key 变了
➔
D
✗ 同上，级联

结果：B、C、D 全部重跑，A 被缓存。关键发现：C 和 D 的 prompt 与上次运行完全相同，但它们的缓存键改变了。这只能意味着键是前缀链式的 — 编辑 B 扰动了所有后续调用。

📈 缓存命中率总结
场景缓存命中率原因
纯顺序，无修改100%确定性链 → 相同的键
编辑某个调用部分从编辑点开始，所有后续键级联变化
并发重排序部分完成顺序不确定性扰动链

⚙ 机制四：parallel / pipeline 调度

parallel 和 pipeline 本身不做任何调度 — 一个单独的信号量（限流器）限制活跃智能体为 min(16, cores-2)。30 个 agent() 调用分两波完成（先 16 个，再 14 个）。

parallel(thunks[])

  • 屏障语义：等待所有完成
  • 保序：结果按输入顺序
  • 异步拒绝 → 该位置 null
  • 同步抛出 → 导致整个运行崩溃
// 实际实现（源自 traces/experiment-05）
return Promise.all(
  thunks.map(t =>
    t().then(v => v, () => null)
  )
)

pipeline(items, ...stages)

  • 无跨阶段屏障：各 item 独立流转
  • 阶段参数：(prev, original, index)
  • 阶段抛出 → 该 item null，跳过后续阶段
  • 其他 item 正常继续
// 每个 item 独立的异步链
items.map((item, i) => (async () => {
  let acc = item
  for (const stage of stages)
    acc = await stage(acc, item, i)
  return acc
})()

实测数据（实验 01）

📊 并发实验证据

parallel：3 个扇出智能体在 2ms 内启动（18:57:32.142 / .143 / .144）。下一个阶段等待最慢的智能体（18.8s）完成后才开始。屏障语义证实。

pipeline：item B 的 stage-2 在 57.010s 开始，早于 item A 的 stage-1 在 58.066s 结束。无跨阶段屏障证实。

并发上限：30 个并行智能体的实测分析显示峰值恰好为 16（主机 32 核 → min(16, 30) = 16），分两波（16 + 14）。

🔌 机制五：可插拔的智能体单元

backend.run(...) 是唯一接触真实智能体的代码行 — 因此"智能体"可以是确定性模拟、Claude API 调用、真实的 CLI 智能体进程，或远程 HTTP 运行器。每种机制对每种后端完全相同；只有单元改变。

后端智能体单元用途
MockBackend合成的确定性结果离线测试 / 保真度门控
AnthropicBackendClaude API 调用 + 工具使用循环直接 API 调用
CliAgentBackend真实本地智能体进程 .claude() / .codex() / .opencode() / .worker('ccz')本地混合舰队
HttpAgentBackend远程智能体（HTTP）云端运行器

这意味着单一编排可以扇出一个混合舰队 — Claude Code 做复杂推理，Codex/OpenCode 做代码编辑，免费的本地 worker 做体力活 — 这是封闭的 Claude-only 引擎做不到的。

§4 完整 API 参考

以下是在工作流脚本中可用的所有原语的完整文档。API 名称（agent/parallel/pipeline/phase/log/budget/workflow）未在公开文档中发布，我们通过运行真实工具并观测行为获得。

agent(prompt, opts?)

派生一个子智能体并返回其结果。

const result = await agent("分析这段代码的安全漏洞", {
  label: "security-review",     // 显示标签（默认 "agent#N"）
  phase: "Review",             // 所属阶段
  schema: {                       // JSON Schema → 结构化输出
    type: "object",
    properties: {
      vulnerabilities: { type: "array", items: { type: "string" } },
      severity: { type: "string", enum: ["low", "medium", "high"] }
    },
    required: ["vulnerabilities", "severity"]
  },
  model: "sonnet",               // 模型别名或完整 ID
  isolation: "worktree",       // 在 git worktree 中隔离运行
  agentType: "workflow-subagent" // 子智能体类型（默认值）
})

parallel(thunks[])

并发执行所有 thunk，屏障（等待全部完成），结果按输入顺序排列。

const results = await parallel([
  () => agent("审查安全性"),
  () => agent("审查性能"),
  () => agent("审查可维护性"),
])
// results[0] = 安全性结果, results[1] = 性能结果, ...
// 如果某个 thunk 异步拒绝 → 对应位置为 null

pipeline(items, ...stages)

每个 item 独立通过所有阶段。无跨阶段屏障 — item A 可能在 stage 3 时，item B 才刚到 stage 1。

const results = await pipeline(
  ["security", "performance", "style"], // items

  // stage 1: 审查 (prev = item 本身, original = item, index = 0,1,2)
  (dim, orig, i) => agent(`审查代码的 ${dim} 维度`, { schema: FINDINGS }),

  // stage 2: 验证每个发现
  (review, orig, i) => parallel(
    review.findings.map(f => () =>
      agent(`验证: ${f.title}`, { schema: VERDICT })
    )
  )
)

phase(title)

设置当前进度组名称。后续 agent() 调用会被归入该阶段。

phase("Scan")              // 后续 agent() 归入 "Scan" 阶段
const flaky = await agent("扫描 CI 日志")
phase("Fix")               // 切换到 "Fix" 阶段

log(message)

输出叙述性进度消息。

log(`已发现 ${bugs.length}/10 个缺陷`)

budget

回合级全局输出 token 计量器。硬上限 — 当 spent() >= total 时 agent() 抛出异常。

budget.total       // number | null（无限制时为 null）
budget.spent()      // 累计输出 token 数
budget.remaining()  // total===null ? Infinity : max(0, total-spent())

// 循环直到预算耗尽
while (budget.total && budget.remaining() > 50000) {
  const r = await agent("发现更多缺陷")
}

workflow(nameOrRef, args?)

内联运行另一个工作流。嵌套限制为恰好 1 层 — 子工作流调用 workflow() 会抛出异常。

const result = await workflow("review-changes")
// 或指定脚本路径：
await workflow({ scriptPath: "./review.js" }, ["arg1"])

args

通过 Workflow 工具调用传入的参数值。

// 调用时: Workflow({ scriptPath: "...", args: ["a.ts", "b.ts"] })
const files = args  // ["a.ts", "b.ts"]

结构化输出（Structured Output）

当 opts.schema 被提供时，引擎注册一个名为 StructuredOutput 的工具，其 input_schema 是用户提供的 schema，并通过 tool_choice:{type:"tool",name:"StructuredOutput"} 强制调用。返回的对象就是 tool_use 块的 .input。

§5 执行模型详解

并发上限

min(16, max(1, cpu_cores - 2)) — 实测在 32 核主机上，30 个并行智能体的峰值并发恰好为 16，分两波执行（16 + 14）。max(1, ...) 确保即使在单核机器上也至少有 1 个并发槽位。官方文档确认"16（在有限 CPU 上更少）"。

生命周期上限

1000 个智能体/运行（硬性上限）。这是一个兜底机制，防止失控循环。

预算机制

budget.spent() 是一个回合级全局输出 token 计量器（主循环 + 所有工作流共享池）。实测值 tokensSpent: 56533 远低于会话的真实 token 使用量 — 所以它只计算输出 token。total=null 时 remaining() === Infinity。

⚠ 预算是尽力而为

agent() 在调用时检查 spent() >= total。它不是严格的逐调用限制 — 一个正在运行的智能体不会在每次调用时被即时截断，N 个并发智能体可能各自通过检查后才累加计数。

模型别名

子智能体 = 完整的 Claude Code 智能体

子智能体不是简单的 LLM 调用 — 它是拥有 Bash、文件系统、技能和 MCP 工具的完整 Claude Code 智能体。探针数据显示一个智能体实际执行了 shell 命令和文件读取操作后才回答。

§7 恢复与缓存机制

日志格式

追加式 JSONL，每行一个 JSON 对象：

{"type":"started","key":"v2:a1b2c3d4e5f6...","agentId":"a001-1"}
{"type":"result","key":"v2:a1b2c3d4e5f6...","agentId":"a001-1","result":{"answer":"..."}}
{"type":"started","key":"v2:f6e5d4c3b2a1...","agentId":"a002-2"}
{"type":"result","key":"v2:f6e5d4c3b2a1...","agentId":"a002-2","result":"[text output]"}

恢复算法

  1. 加载前次运行的 result 事件到 Map<key, result>
  2. 重新执行脚本 — 每个 agent() 调用重新计算 key
  3. 如果 key 匹配 → 直接返回缓存结果，不调用后端，不占用并发槽
  4. 如果 key 不匹配 → 正常执行（调用后端、写入日志）

三种缓存场景

✓ 纯顺序，无修改

100% 缓存命中。确定性链产生相同的键序列。所有 agent() 调用直接返回缓存。

⚠ 编辑某个调用

级联效应：从编辑点开始，所有后续调用的 key 改变。编辑点之前的调用缓存命中，之后全部重跑。

⚠ 并发重排序

parallel/pipeline 的完成顺序不确定，扰动了下游的链，导致部分缓存命中。

§8 错误处理与边界情况

parallel vs pipeline 的非对称性

这是最令人惊讶的发现之一。两种原语的错误处理行为不对称：

情况parallelpipeline
空输入[][]
异步拒绝（如 agent 错误）对应位置 null该 item null，跳过后续阶段
同步抛出导致整个运行崩溃该 item → null
多个同时出错各位置独立 → null各 item 独立 → null
⚠ parallel 的同步抛出会崩溃

文档说抛出的 thunk "解析为 null"。但第一个探针在 thunk 中放入同步 throw — 整个工作流崩溃了。只有异步拒绝变为 null。

原因：t().then(v => v, () => null) — t() 中的同步抛出在 .then 能捕获之前就逃逸了 .map 回调。

而 pipeline 中每个阶段在异步 per-item 链内运行，同步抛出被包含为该链的拒绝 → null。这种不对称性是真实存在的。

🔎 嵌套限制（experiment-05）

workflow() 嵌套限制为恰好 1 层。顶层工作流可以调用 workflow()，但子工作流调用 workflow() 会抛出精确的错误消息：

// 顶层 workflow
const child = await workflow("child")  // ✓ 正常

// 子工作流内部：
await workflow("grandchild")  // ✗ 抛出异常！
// Error: "workflow() cannot be called from within a child workflow —
// nesting is limited to one level. Inline the inner script or call its agents directly."

§9 经典组合模式

🔍 审查 → 验证（Review → Verify）

最经典的工作流形状：多个维度并行审查 → 每个发现独立验证 → 过滤确认的。

const DIMENSIONS = [
  { key: "bugs", prompt: "查找正确性缺陷" },
  { key: "security", prompt: "查找安全漏洞" },
  { key: "performance", prompt: "查找性能问题" },
]

const results = await pipeline(
  DIMENSIONS,
  // stage 1: 每个维度并行审查
  d => agent(d.prompt, { label: `review:${d.key}`, phase: "Review", schema: FINDINGS }),
  // stage 2: 每个发现并行验证
  review => parallel(
    review.findings.map(f => () =>
      agent(`对抗性验证: ${f.title}`, { label: `verify:${f.file}`, phase: "Verify", schema: VERDICT })
    )
  )
)
const confirmed = results.flat().filter(v => v?.isReal)

📚 多模态研究

const sources = await parallel([
  () => agent("按容器搜索", { phase: "Find" }),
  () => agent("按内容搜索", { phase: "Find" }),
  () => agent("按实体搜索", { phase: "Find" }),
])
const deepRead = await pipeline(
  sources.flat(),
  src => agent(`深入阅读: ${src.url}`),
  analysis => agent("综合分析", { schema: REPORT })
)

🔄 循环直到耗尽（Loop-until-dry）

const seen = new Set(), confirmed = []
let dry = 0

while (dry < 2) {                             // 连续 2 轮无新发现则停止
  const found = (await parallel(
    FINDERS.map(f => () => agent(f.prompt, { phase: "Find", schema: BUGS }))
  )).flat().filter(Boolean).flatMap(r => r.bugs)

  const fresh = found.filter(b => !seen.has(key(b)))
  if (!fresh.length) { dry++; continue }
  dry = 0
  fresh.forEach(b => seen.add(key(b)))

  const judged = await parallel(
    fresh.map(b => () => parallel(
      ["correctness", "security", "repro"].map(lens => () =>
        agent(`通过 ${lens} 视角判断 "${b.desc}"`, { phase: "Verify", schema: VERDICT })
      )
    ))
  )
  confirmed.push(...judged.filter(v => v.real))
}

💰 循环直到预算耗尽（Loop-until-budget）

const bugs = []
while (budget.total && budget.remaining() > 50000) {
  const result = await agent("查找代码库中的缺陷", { schema: BUGS_SCHEMA })
  bugs.push(...result.bugs)
  log(`已发现 ${bugs.length} 个，剩余 ${Math.round(budget.remaining()/1000)}k token`)
}

§14 诚实的局限性

⚠ 诚实的边界

本复现再现了可观测行为，而非 Anthropic 的内部实现。

方面说明
v2 哈希原像未公开 — 我们匹配形状和语义，而非字节
预算作用域我们的引擎是运行范围（run-scoped），真实引擎是回合全局（turn-global）
vm 沙箱仅限 Node.js — 不是安全边界，仅匹配可观测 API
子智能体建模为可插拔单元，而非嵌入 Claude Code 本身
恢复范围官方仅支持同会话恢复；我们的跨会话日志恢复是故意的超集
成熟度v0.x — 新项目，未在大规模场景中验证
社区最小社区 — 新项目
UI / 可观测性无 Web UI、无 LangSmith/AgentOps 集成
Python API仅有 TypeScript API
